# Change: Refactor built-in reviews to use YAML config + pure markdown prompts

**Supersedes**: `add-built-in-reviews` (archive that change before applying this one)

## Why
The current built-in review implementation bundles frontmatter (settings) inside the markdown prompt and uses Bun-specific text imports (`with { type: "text" }`). This couples runtime settings to the prompt content, requires Bun-specific syntax, and bypasses the existing YAML review config path. Users cannot customize settings (like `num_reviews`) for a built-in review without abandoning it entirely.

The `init` command currently references `built-in:code-quality` directly in `config.yml` entry points, meaning there is no review file for the user to inspect or customize settings on.

## What Changes
- Built-in review prompts become **pure markdown** (no frontmatter). They contain only the prompt text.
- YAML review config files (`.yml`/`.yaml`) gain a new `builtin` attribute (mutually exclusive with `prompt_file` and `skill_name`) that loads a built-in prompt by name.
- The `init` command generates `.gauntlet/reviews/code-quality.yml` (a YAML review config referencing the built-in) instead of putting `built-in:code-quality` directly in `config.yml`.
- The `config.yml` generated by `init` references `code-quality` (the file-based review) instead of `built-in:code-quality`.
- The built-in code-quality prompt is updated to match the existing `.gauntlet/reviews/code-quality.md` but without the Documentation focus area (project-specific docs should not be in a generic built-in).
- The `built-in:` prefix syntax in `config.yml` entry points is removed. Built-in reviews are now always accessed via YAML review config files.

## Alternatives Considered
- **Keep `built-in:` prefix in entry points**: Allows inline references without review files, but prevents users from customizing settings per-project. The YAML config approach gives users a visible, editable file with full control over settings.
- **Frontmatter in built-in prompts as defaults**: Built-in prompts could keep frontmatter as default settings that YAML overrides. Adds complexity with two-layer merging. Simpler to have one source of truth for settings (the YAML file) and one for prompt content (the built-in markdown).

## Impact
- Affected specs: `review-config` (MODIFIED — replace `built-in:` prefix with `builtin` YAML attribute), `init-hook-install` (MODIFIED — init generates YAML review file)
- Affected code:
  - `src/built-in-reviews/code-quality.md` — remove frontmatter, update prompt content
  - `src/built-in-reviews/index.ts` — simplify to return raw markdown (no gray-matter parsing, no schema validation)
  - `src/config/schema.ts` — add `builtin` to `reviewYamlSchema`, update mutual exclusivity rules
  - `src/config/loader.ts` — handle `builtin` attribute in YAML review loading, remove `built-in:` prefix resolution from entry points
  - `src/config/types.ts` — remove `isBuiltIn` field (no longer needed; review is file-based with built-in prompt content)
  - `src/commands/init.ts` — generate `.gauntlet/reviews/code-quality.yml`, update `config.yml` to reference `code-quality`
  - `test/config/loader.test.ts` — update built-in review tests
